## 1. 使用者族群 (User Segments)

牙醫診所網站的使用者不是只有一種，他們的痛點、決策方式、進站動機差異很大。如果一視同仁，做出來的網站就會誰都不滿意 — 這就是講義 P.39「工程導向思維」的陷阱：以為問題已經被定義好。

| 代號 | 族群 | 進站關鍵字 | 商業重要性 |
|---|---|---|---|
| **U1** | 急性疼痛患者 | 「牙痛」「附近牙醫 急診」 | 流量大、轉換快 |
| **U2** | 家庭主要照顧者 | 「兒童牙科」「植牙費用」 | 客單價最高、決策最謹慎 |
| **U3** | 美學需求顧客 | 「冷光美白」「隱形矯正」「貼片」 | 高毛利、看品牌與案例 |
| **U4** | 診所員工 / 管理員 | (不從外面進來) | 後台使用者，常被忽略卻是系統能跑的關鍵 |

## 2. Empathy Map（每個族群一張）

格式遵循 NN/g 的 Empathy Map：**Says / Thinks / Does / Feels**（講義 P.17–23）。

### 🤕 U1 — 急性疼痛患者

| 象限 | 內容 |
|---|---|
| **Says (說)** | 「最快什麼時候能看？」「現在還有開嗎？」「自費還是健保？多少錢？」 |
| **Thinks (想)** | 「希望不要等太久」「不要被推銷一堆療程」「希望醫師看起來專業」 |
| **Does (做)** | 用手機 Google「附近 牙醫 急診」、開 Google Maps 看評論、找門牌、打電話 |
| **Feels (感)** | 焦慮、痛、不耐煩；想盡快得到「能不能看」的明確答案 |
| **表面需求** | 馬上預約、知道地點電話 |
| **深層恐懼** | 「跑去結果關門 / 沒名額」浪費時間 |

### 👨‍👩‍👧 U2 — 家庭主要照顧者

| 象限 | 內容 |
|---|---|
| **Says** | 「小孩第一次看牙不要嚇到他」「我爸植牙這價錢合理嗎」「能用健保嗎」 |
| **Thinks** | 「這個醫師可靠嗎？學經歷？」「之前有人推薦的好像比較好」「療程要分幾次？多久？」 |
| **Does** | 比 3–5 家診所、看 Google 評論、問 LINE 群朋友、看官網醫師簡介、查資歷 |
| **Feels** | 謹慎、責任感、想做出「對家人最好」的決定；對被當盤子有警戒 |
| **表面需求** | 看醫師資歷、看價格、看療程說明 |
| **深層恐懼** | 信任錯醫師 / 被過度治療 |

### 💄 U3 — 美學需求顧客

| 象限 | 內容 |
|---|---|
| **Says** | 「之前的案例可以看嗎」「療程要幾次」「會痛嗎」「能分期嗎」 |
| **Thinks** | 「網路上有沒有人在 IG 推薦這家」「環境拍起來漂不漂亮」「醫師美感如何」 |
| **Does** | 看官網「案例」、IG、追蹤醫師個人帳號、看環境照片、私訊或填表諮詢 |
| **Feels** | 期待 + 緊張；很在意「視覺品質」與「同溫層口碑」 |
| **表面需求** | 案例照、價格區間、醫師美感與經歷 |
| **深層恐懼** | 做完效果不如預期、被笑「智商稅」 |

### 👔 U4 — 診所員工 / 管理員

| 象限 | 內容 |
|---|---|
| **Says** | 「為什麼又有人重複預約」「我下午要更新一下醫師休診」「最近預約查不到」 |
| **Thinks** | 「介面別讓我點 10 下才完成一件事」「不要動不動就掛掉」「資料不要弄丟」 |
| **Does** | 看當日預約清單、確認 / 取消預約、編輯休診表、回覆聯絡表單 |
| **Feels** | 忙、煩；最怕「系統卡住但病患在前面等」 |
| **表面需求** | 好用的後台 |
| **深層恐懼** | 系統當機影響當天看診流程 |

## 3. 從 Empathy Map 收斂出 User Story

格式（講義 P.26）：
> **身為 \_\_\_（角色），我想要 \_\_\_（需求描述），這樣才能 \_\_\_（使用者價值）。**

注意「壞範例」（P.27、29、30）：不能寫「身為使用者，我想要更好的體驗，這樣才能提升回訪率」— 沒有具體角色、沒有可觀察需求、沒有使用者價值。

### MVP（Phase 3–5 要做完的）— 用 ★ 標示

| ID | User Story | 對應族群 | 優先級 |
|---|---|---|---|
| **US-01** ★ | 身為**牙痛急著找牙醫的病患**，我想要**在首頁就看到「今天還能不能看 / 電話 / 地址」**，這樣我才能**立刻決定是否前往** | U1 | P0 |
| **US-02** ★ | 身為**牙痛急著找牙醫的病患**，我想要**用 3 步以內完成線上預約**，這樣我才能**不用打電話也能掛號** | U1 | P0 |
| **US-03** ★ | 身為**幫家人挑醫師的照顧者**，我想要**看到每位醫師的學經歷與專長**，這樣我才能**判斷該掛哪位醫師** | U2 | P0 |
| **US-04** ★ | 身為**幫家人挑醫師的照顧者**，我想要**看到清楚的療程價格區間與是否健保給付**，這樣我才能**避免事後價錢嚇到** | U2 | P0 |
| **US-05** ★ | 身為**診所櫃台**，我想要**在後台看到今日預約清單並能標記已到 / 未到**，這樣我才能**掌握當日來客** | U4 | P0 |
| **US-06** ★ | 身為**診所管理員**，我想要**新增 / 編輯醫師與服務項目**，這樣我才能**不靠工程師也能更新網站** | U4 | P0 |
| **US-07** ★ | 身為**任一族群病患**，我想要**收到預約成功通知（Email）**，這樣我才能**確認預約有效並有記錄** | U1, U2, U3 | P0 |

### 後續迭代（Phase 7–8 之後可以加）

| ID | User Story | 對應族群 | 優先級 |
|---|---|---|---|
| US-08 | 身為**美學療程客**，我想要**看到實際案例的前後對比照**，這樣我才能**評估療程效果** | U3 | P1 |
| US-09 | 身為**任一族群病患**，我想要**查詢與取消我的預約**，這樣我才能**改變計畫時不卡死** | All | P1 |

| US-12 | 身為**診所管理員**，我希望**系統有監控、出事時能即時知道** | U4 | P1 (Phase 8) |

## 4. Acceptance Criteria (AC) — 以 MVP US-02 為例

格式（講義 P.33）：**前置情境 / 觸發條件 / 期望結果**。AC 必須具體、可驗證、有邊界，**不能等於實作方式**（P.36）。

### US-02：線上預約

| 前置情境 | 觸發條件 | 期望結果 |
|---|---|---|
| 使用者進入預約頁面 | 點選「線上預約」 | 顯示醫師清單與可預約時段 |
| 使用者已選擇醫師與時段 | 點「確認預約」並填妥姓名 / 電話 / 健保卡號 | 系統儲存預約並回傳預約編號 |
| 預約成功 | — | 系統發送 Email 通知病患（內容含醫師、時段、診所地址） |
| 使用者選的時段已被別人預約走 | 在同一時段送出 | 顯示「該時段已滿，請選其他時段」並不寫入資料庫 |
| 預約成功 30 分鐘內 | 預約還未開始 | 使用者可以用「預約編號 + 電話末四碼」查詢預約 |

### US-05：後台今日預約

| 前置情境 | 觸發條件 | 期望結果 |
|---|---|---|
| 員工已登入後台 | 進入「今日預約」 | 列出當日所有預約，依時段排序 |
| 員工點某筆預約的「已到」 | — | 預約狀態變為「已到診」，列表即時更新 |
| 員工點某筆預約的「未到」 | — | 預約狀態變為「未到」並從待處理清單移除 |
| 員工試圖看其他日期 | 點日期選擇器 | 可切換日期，URL 帶 query 參數讓重整能保留 |

## 5. MVP 範圍定義（Phase 3 結束時要有的東西）

呼應講義 P.49「MVP = 用最低成本驗證核心價值」、Agile P.26 (Skateboard → Car)。

### ✅ MVP 必做
1. **首頁**：診所資訊、今日是否營業、電話、地址、Google Map embed
2. **醫師列表頁** + **單一醫師頁**：照片、學經歷、專長
3. **服務 / 療程頁**：項目、價格區間、健保註記
4. **線上預約**：US-02 的 happy path（暫時不做付款）
5. **預約 Email 通知**：US-07
6. **後台**：US-05、US-06（最簡單的 CRUD）
7. **手機 RWD**：US-11

### ❌ MVP 不做（之後 iteration 再加）
- 線上付款 / 訂金
- 案例照片牆 / IG 整合
- 多語系（i18n）
- 簡訊通知
- 病歷整合
- AI 諮詢機器人

> **為什麼這樣切？** 講義 P.49 的核心：MVP 必須「保持簡單、不必完美，但必須完整且可用」。我們先讓一個族群（急性病患 + 照顧者）能完整跑完「進站 → 找資訊 → 預約 → 收 Email」的路徑，這就是「能滑的滑板」(P.50)。

## 6. 非功能性需求 (NFRs)

對應講義 ntu_app_arch P.11、System Architecture：

| NFR | 目標（MVP 階段） | 對應 Phase |
|---|---|---|
| **可用性 (Availability)** | 99% — 全月最多 7 小時 downtime（不含計劃維護） | Phase 7 |
| **效能 (Performance)** | 頁面首屏 < 3s、API P95 < 500ms | Phase 8 |
| **可擴充性 (Extensibility)** | 加新服務項目 / 醫師不需要動 code | Phase 3（分層架構） |
| **安全性 (Security)** | HTTPS、密碼雜湊、後台 JWT、不存信用卡 | Phase 4、Phase 7 |
| **可觀察性 (Observability)** | 結構化 log + /health + 4 Golden Signals | Phase 8 |
| **可移植性 (Portability)** | 同一個 image 可跑 dev/stage/prod | Phase 4、5 |

## 7. Definition of Done（每個 Story 都要過這一關）

對應講義 Agile P.48「Good Backlog」與 12-Factor SDLC：

- [ ] 程式碼通過 lint + type check
- [ ] 有對應的單元測試 / API 測試
- [ ] AC 的所有情境都驗過
- [ ] 沒有 secret 寫死在 code 裡
- [ ] Log 用 stdout、有對應的結構化欄位
- [ ] PR 描述包含 What / Why / How to test
- [ ] CI 全綠
- [ ] 在 staging 環境跑過（Phase 7 之後）
